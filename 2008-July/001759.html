Here is one: It brings no real benefit neither transparency of how the
things happen.

Having a simple interface as seen above is nice and transparent to the
user. And having the possibility to specify expert settings in a
hw-independent way is also nice for the expert-user.

But if you like to have a set of tables there is *no* reason, why these
tables need to be kept in Kernelspace. We should keep out things that
only increase complexity without having a real necessary functionality.
This table handling is dumb stuff that also can be handled via some
scripts in userspace - scripts you would need anyway for your CAN
interfaces startup.

Having tables in Kernelspace is avoidable effort and brings no real
additional benefit.

I think Wolfgang should go for his suggested solution.
