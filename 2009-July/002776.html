Oliver Hartkopp wrote:
> Hi Wolfgang,
> 
> after reading a bit in the CC770 spec i wonder, if having the CC770 additional
> functions to be 'on' by default is the correct approach - especially because
> the CTRL_EAF bit should not be set to '1' when having the AN82527.

I read that as well, but as the bit is read-only on AN82527 I do not
understand why and the method of testing the EAF works fine, as you
demonstrated on your hardware.

> I assume most hardware out there uses the AN82527 these days ...

This will change. The AN82527 chip is not any more available, already
since a few years.

> I would suggest to add the configuration bits for EAF and the msgobj15_eff
> somewhere into struct cc770_priv and add 'eaf' or 'cc770_eaf' and
> 'msgobj15_eff' as module parameter array(!) to cc770_isa.

The msgobj15_eff should be per device, of course, but we do not have a
way yet to set it on a per device base for non-legacy drivers.

> Although 'cc770_eaf' should be off (== 0) by default.
> 
> What do you think about this?

Well, I think it's a real benefit using the message object 15 for all
frame types due to the double buffering. Anyhow, the user will either
use it for SFF *or* EFF which makes it also unnecessary to fiddle with
the "msgobj15_eff" option. And as the name suggests, this driver is
preliminary for the CC770, with a compatibility mode for the AN82527.
Well, I still prefer using the additional features for the CC770 as
default until problems show up.

Wolfgang.
