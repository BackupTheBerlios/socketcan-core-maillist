Hi all,

after discussing some points with Urs i'd like to pull back my RFCs 
regarding can_frame.[priv|origin] :

https://lists.berlios.de/pipermail/socketcan-core/2007-October/000769.html
https://lists.berlios.de/pipermail/socketcan-core/2007-October/000772.html

Reasons:

- The discussed 'meta' information should not be part of the 'data' CAN 
frame as the struct can_frame should just contain the CAN frame to keep 
a simple and clear interface.

- When there is a reason to know about sending problems the 'error 
frame' has the answer for it like ICMP in the Internet Protocol. 
Although there is no feedback on per-frame level there should be an 
appropriate error handling.

When anyone would like to use the gap in the struct can_frame, he could ...

- change the struct can_frame for his personal purpose (yeah it's OSS :)
- define an own struct overlay to access the gap via unions or pointer 
castings

Sorry for the noise!

Oliver.
