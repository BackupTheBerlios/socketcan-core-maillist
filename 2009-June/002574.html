Wolfgang Grandegger wrote:

> 
> I'm not really happy with that solution. Sooner than later may have
> problems with other include files, e.g. netlink.h. This will happen if
> we want to extend the interface, say adding a new member to can_priv in
> dev.h, or a new netlink message to netlink.h. Furthermore, the name
> "dev_compat.h" would no longer be appropriate. I would really *avoid*
> sharing header files between kernel and the external module by using
> different path names pointing to them. The path could then be corrected
> by strip-src.

Convinced!

When providing source code that differs from the mainline - why not provide
separate include files that functionally do the same thing but do not clash
with the mainline stuff, right?

I tried this in socketcan/branches/netlink/kernel/2.6 :

svn cp include/linux include/socketcan

grep -rl linux\/can . | grep -v svn | grep -v linux | grep -v networking |
xargs sed -i -e 's/linux\/can/socketcan\/can/'

Works excellent ;-)
