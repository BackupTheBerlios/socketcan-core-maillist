Hi Wolfgang.

On Mon, Dec 03, 2007 at 10:52:24AM +0100, Wolfgang Grandegger wrote:
> This patch adds preliminary SYSFS support for CAN devices to "dev.c". It
> now supports setting and getting of CAN device properties via both, SYSFS
> files and IOCTL requests by using common access functions. If we decide
> to abandon and remove IOCTL request, SYSFS support should be move to a
> seperate file, though.

If this work is presented to netdev the first reply would be to avoid
two interfaces solving the same purpose - and as sysfs is the preferred
interface these days this will be the requested interface.

As CAN support in the kernel is considered new at netdev (only recently
merged) then I do not see them buying into backward compatibility reasons.
So the way forwards seems to be to go only for sysfs (and take the
issue sooner with existing userland).

> This is a RFC and it rises various questions:
> 
> - do we abandon IOCTL requests because there is no chance to get them
>   into mainline? Already now or later?

Already covered above.

> 
> - Location of SYSFS files: Is "/sys/class/net/can*" OK or do we need
>   to define a separate SYSFS bus and class?
> 
> - Naming issues, e.g. a prefix "can_" for all CAN properties like
>   can_state.

Try to raise these questions on netdev.
If the can mailing list is moderated then do not cc: the can mailing list.
It seems to raise the blood pressure for people.
