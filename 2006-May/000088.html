The error frame is definitely NOT controller specific. But we should think
about having it only that size to cover the current needed size (which is
6). I just wanted the last two bytes left blank for 'bad hacks' which should
not go into a common specification for an error-frame. Or we just say:
data[6..7] may contain controller specific data (which can not be covered by
the generic error description in data[0..5]).

What do you think on this?

> 
> And where definition of tx(rx?) timeout(s)?
> 

The TX-timeout is someting that is recognized by a standard netdevice-driver
technique. Maybe it is a good idea to inform the user on this problem:

#define CANERR_CRTL_TX_TIMEOUT  0x10 /* timeout sending a CAN frame */

But the RX-timeout is something really application specific. So it has
nothing to do in an error frame. The Broadcast-Manager (BCM) can monitor
(cyclic) CAN-Messages for your application and can send a notification on
RX-timeouts, if you like.
