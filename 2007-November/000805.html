> Exactly! E.g. we now have a defined place where to put the drivers in
> drivers/net/can and we have a Kconfig file there that can be extended

> As you stated some more times before the current drivers are not kernel
> style. They are grown from the first CAN netdev implementations and work
> fine so far (at least for me!?). But after getting an insight into the
> reviews on the netdev ML, we really need to check for current netdevice
> interface APIs, PHYs, platforms, debugging methods and so on. And indeed
> i'm a complete newbie in PHYs and platform stuff :-/

ack

> From what i've seen and understood right now, the new AT91 driver from
> Hans-Juergen looks very nice to me.

We have a reworked SJA1000 driver in the pipeline.

> Should the *new* created netdrivers support older kernels (like 2.6.16)
> or should they start from 2.6.24 and newer as long as they are not part
> of the Kernel? The *current* (old) drivers in the SVN could stay there
> analogue to the CAN core both supporting also older kernels.

I'd very much prefer patches in svn, which is much easier to rebase to
several kernels. At the moment we are again maintaining our own
socket-can patch queue, because you are already on 2.6.24-rc and we have
to ship customer kernels on top of 2.6.23. Drivers are orthogonal
enough, so it should be no problem to keep them in quilt stacks.

> Should we use a different revision control system or a git for the CAN
> netdriver development? SVN is fine for me as it works through
> http-proxies.

Please, let's stay with either the current variant or that one outlined
above. I know that everyone thinks git is the best rcs since sliced
bread, but it is not the solution for everything and creates a bunch of
new problems. Let's please focus on the code, not on tools.

> Should we create a branch for driver development? And if we go for a
> different system like git for the drivers - who could host & maintain
> it? Questions over questions ...  The SVN and the possibility of
> external(!) compilation turned out to be a very good approach to me in
> the past.

We'll probably stay with 2.6.23 until socket-can is finally merged
(unless there is some other customer requirement). So for us it would
especially be interesting to see how to get the missing technological
things in place, like the still mostly undiscussed question of bittime
tables.
