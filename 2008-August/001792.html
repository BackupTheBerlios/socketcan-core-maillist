>> Well, I see. But I would not move the CDR and OCR defines to

>  But with the struct platform_data
> definition inc OCR_*  and ODR_* in include/linux/can/sja1000_platform.h,
> out-of-tree building is still possible, just like before. But then you
> have to hard code the hex values in the platform specific code (which is
> in the kernel).
> On the other hand if you integrate the berlios patches into you kernel
> patchstack or the driver finally ends up in mainline you can use the

Sorry but this does not convince me at all.

The clock and the OCR/ODR stuff is board specific and can perfectly be
defined in the <board>.c code, as this would be the ONLY place of
reference. No way for include/linux/can/* ...

This would be a big step into a superfluous maintainance problem.

Anyway: If you think about handling OCR/ODR we should make the correct
step and define some kind of standard PHY interface. There are CAN
transceivers (PHYs like the TJA1054 or TJA1041) that allow wake-up on
CAN traffic and also to switch off the ECUs Power Supply by setting
several pins on the transceiver.

Maybe this is board-specific - or maybe we should define (or use) some
PHY interface.

But i cannot see any reason why to put this board specific stuff into
include/linux/can/* unless you come out with a really good reason ...
