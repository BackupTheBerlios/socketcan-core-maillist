Chris Elston wrote:
> I can't speak as to TTCAN requirements, my client has their own protocol
> which requires single-shot. My understanding of the requirement is that
> it needs to be possible to mark individual frames as single-shot.  I
> don't think a per-bus setting will allow us to synchronise which frame
> should  be sent in this mode.
>
> I guess the alternatives are to have either a socket configuration
> option to set single-shot mode, or to add it to the frame structure.
> That still leaves the problem of how we get that information to the
> driver in the skb.  Ideas?
>   

The CAN-frame structure describe the CAN-frame and not the mode how this 
could be sent on the bus - so we should leave it, as it is.
I could image to add a new socket-option like CAN_RAW_ONESHOT (or 
something that hit's it better - any suggestions?).

Currently we use skb->pkt_type to tell the driver wether the frame 
should be looped back or not. So we could think about a different 
skb->pkt_type to say 'oneshot' to the driver.

So it's a 5-minutes work to make this a socket-option, but i would tend  
to integrate that *after* the current stuff is mainlined. I could sent 
you a patch for testing and review by the others , if it's ok for you.

Oliver
