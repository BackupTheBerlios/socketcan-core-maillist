Uwe Koppe wrote:
>> Well,
>> I think that's not general enough. It might be OK for CAN controllers
>> using just one or two Oszillator frequencies like the SJA1000, but for
>> on-chip controllers like MSCAN on the MPC5200 it can be almost any
>> frequency (I know of 25MHz, 33MHz, 33,333333 MHz, 66 MHz, 132 MHz).
> 
> I checked the datasheet of the MPC5200 and yes, these type of
> controllers are causing trouble: you can only select between
> IP bus clock and oscilliator clock you you have only a prescaler
> of 64 max. Running the system with 132 MHz you can never achieve
> bitrates below 86 kBit/s. In addition, you can't realize the standard
> values I mentioned without an error.

Yes, normally the oscillator clock should be used but due to a chip bug,
it's not possible on Rev. A CPUs. It will always select the IP bus
clock. Nevertheless, we should fix that for Rev.B CPUs.

> Best CAN clock frequencies are always a multiple of 8 Mhz.

Hm, then it might be wise to use a 32MHz oscillator (instead of the
usual 33MHz) for the MPC5200 board.

> The "best guess" calculation leads to unknown sample points and
> inaccurate bitrates. This will cause error frames in the network and my
> experience is that "normal" users are not able to handle these issues.

OK, I think we must live it.

Wolfgang.
