Hi Hans,

Once the AT91 patch for the files in arch/arm/mach-at91/ gets applied to the kernel proper, the confusion will probably go away.  As I checked out the SocketCAN trunk, I just didn't see the relevant data structure at first.

On a more generic note, should struct at91_can_data be changed to something that is more implementation-independent?  I've seen other drivers (notably the Analog Devices adxl34x which I've been integrating lately) that have their own struct adxl34x_data defined with the driver, which specific boards can then choose to use.  At the least, it keeps their driver code independent from any changes in the board files and allows the code to compile without errors.
