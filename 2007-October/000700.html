Chris Elston wrote:
> Marc,
> 
>>> Does the socketcan API provide a way to indicate that a CAN packet
>>> should be transmitted in 'single-shot' mode.  My CAN controller has
>>> support for this, and my customer has a requirement to use it, but I
>>> can't see any support for it in the socketcan API.  I need a way to
>>> signal to the kernel driver I'm writing that a packet should be
>>> single-shot, any ideas?
>> Can you please explain the concept of "signle-shot", I'm not familliar 
>> with it.
> 
>>From the MCP2515 datasheet:
> 
> "3.4 One-Shot Mode
> 
> One-shot mode ensures that a message will only attempt to transmit one
> time.  Normally, if a CAN message loses arbitration, or is destroyed by
> an error frame, the message is retransmitted.  With One-shot mode
> enabled, a message will only attempt to transmit one time, regardless of
> arbitration loss or error frame.
> 
> One-shot mode is required to maintain time slots in deterministic
> systems, such as TTCAN."

The SJA1000 seems also to support this feature. Controller specific 
setting could be done with the IOCTL request SIOCSCANCTRLMODE. So far we 
have defined:

   cat include/linux/can(ioctl.h
   ...
   #define CAN_CTRLMODE_LOOPBACK   0x1
   #define CAN_CTRLMODE_LISTENONLY 0x2

And it makes sense to add:

   #define CAN_CTRLMODE_SINGLESHOT 0x3

Wolfgang.
