Marc Kleine-Budde <mkl at pengutronix.de> writes:

> Please keep in mind, that in 2.6 days kernel-headers are not used
> anymore in userspace. There are special sanitized headers (e.g.[1]) for
> the userspae. We've put out can userspace headers into our canutils
> package [2].

That's something I've always disliked/misunderstood.  When the kernel
exports functionality to user space it should provide the header file
for the user space to include.  Why is there an extra package
containing these include files.  This is an almost sure way to have
header files which are out of date or don't match the current kernel.
If the interface would be completely stable, then why is there
linux-libc-headers-<version> where for <version> there is not only 2.4
and 2.6 (which I would expect but which I would also put into the
normal kernel sources), but 2.6.11.{0,1,2} but no current 2.6.16?

I must admit that I didn't had the time to follow the discussion on
LKML on this topic.
