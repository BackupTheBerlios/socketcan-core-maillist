Robert Schwebel wrote:
> On Thu, Oct 19, 2006 at 09:19:01AM +0200, Wolfgang Grandegger wrote:
>> So far I understood, that it's mainly useful for debugging.
> 
> Well, for debugging you will be fried anyway, at least with more
> performant CAN controllers. For example with nioscan you have 32 send
> buffers in the hardware and the software queue is pushing packets
> therein, as long as there is space left. So even if you know the send
> order on the lowest software level, it wouldn't necessarily be the same
> order as on the wire. 

- Does that controller push out frames in an unpredictable manner?
- Does it report transmitted frames or only an empty tx queue?
- Does it force you to use queueing?

And even if we cannot guarantee the precise order of frames on the wire
vs. those on the virtual local bus for every hardware, we should at
least try to couple the local delivery with the bus state. I.e. if you
pull the real CAN cable, no loopback should be performed.


That's the model for loopback between distributed, node-agnostic
applications, so to say a virtual local CAN bus between them. That
feature should be controllable by the application, because not all of
them need it (to express it neutrally).

Debugging is for me a different story. If you attach a packet sniffer,
you are typically interested in gaining the same behaviour like with
Wireshark on Ethernet. That means log outgoing frames when they are
queued, independent of the loopback setting of some CAN application. See
ETH_P_ALL vs. IP_MULTICAST_LOOP.
